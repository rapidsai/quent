# Decision: Cross-Model Validation via Rust's Type System

## Context

Model items reference each other: FSM states declare resource usages, usages
reference resource types, resources have typed capacities. These
cross-references must be validated to catch errors early (e.g., referencing a
nonexistent resource, using the wrong capacity type).

The options considered were:

1. **Proc macro validation**: a macro (module-level or a separate validation
   pass) inspects all items and checks cross-references.
2. **Rust's type system**: cross-references are expressed as generic type
   parameters with trait bounds. The compiler validates them.

## Decision

Rely on Rust's type system for cross-model validation.

## Rationale

### Usage<T: Resource> enforces resource references

The `Usage<T>` type requires `T: Resource`:

```rust
pub struct Usage<T: Resource> {
    pub resource_id: Uuid,
    pub capacity: T::CapacityValue,
}
```

The `Resource` trait and its associated `CapacityValue` type are generated by
the resource annotations (`#[quent::memory]`, `#[quent::processor]`, etc.):

```rust
// Generated by #[quent::memory] on WorkerMemory
impl Resource for WorkerMemory {
    type CapacityValue = MemoryCapacity; // { used_bytes: u64 }
}
```

If a state declares `Usage<Foo>` and `Foo` is not a resource, the compiler
emits:

```
error[E0277]: the trait bound `Foo: Resource` is not satisfied
```

This works across crate boundaries without any macro coordination.

### Capacity types are enforced by associated types

`T::CapacityValue` resolves to the concrete capacity struct for the resource.
A memory resource yields `{ used_bytes: u64 }`, a processor yields `()` (unit
resource), a custom resource yields its declared capacity struct. The
application code must construct the correct capacity value; type mismatches are
compile errors.

### FSM-internal validation is handled by the proc macro

The `#[quent::fsm]` macro validates the transition graph in isolation:

- All states referenced in transitions must exist as types with the
  `#[quent::state]` annotation (checked via a generated marker trait).
- Every state is reachable from entry.
- Every state can reach exit.
- No transitions leave exit.

These are structural properties of a single FSM and do not require cross-item
visibility beyond type resolution.

### Tradeoff

Rust compiler errors for trait bound failures can be cryptic. The proc macros
should generate `compile_error!` messages where possible to provide clearer
diagnostics. For example, if a state struct is missing `#[quent::state]`, the
proc macro on the FSM can detect this (the state type won't have the `State`
marker trait) and emit a targeted error message rather than letting the raw
trait bound failure surface.
