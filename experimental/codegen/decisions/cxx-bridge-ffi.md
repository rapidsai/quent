# Decision: CXX Bridge for C++ FFI, No Separate C++ Runtime

## Context

The generated C++ instrumentation API needs to emit events that the Rust
instrumentation backend processes (buffering, exporting). The question is how
C++ code communicates with Rust: via a wire format (serialization across a
process or network boundary), via a C ABI, or via a higher-level FFI mechanism.

A separate question is whether a standalone C++ runtime library is needed for
event buffering, async export, and context management.

## Decision

Use CXX bridge for in-process FFI. The Rust instrumentation backend is the
runtime. There is no separate C++ runtime library.

## How it works

The codegen generates Rust `#[cxx::bridge]` modules that expose observer types
and their methods to C++. CXX generates the corresponding C++ headers. The C++
application includes these headers and calls observer methods directly.

Event flow:

```
C++ application code
  ↓ calls observer methods
CXX-generated FFI boundary
  ↓
Rust observer implementation
  ↓ constructs Event<T>, calls push_event()
Rust EventSender → event channel
  ↓
Rust exporters (collector, ndjson, msgpack, postcard, ...)
```

No serialization crosses the FFI boundary. Arguments are passed directly via
CXX's type mappings (integers, strings, UUIDs, structs). Serialization happens
downstream in the exporter layer, entirely in Rust.

## What the codegen produces

**Rust side (generated):**

- `#[cxx::bridge]` module per observer with factory function and methods
- Observer struct wrapping an `EventSender<AppEvent>`
- Method implementations that construct `Event<AppEvent>` and push them
- Event enum variants corresponding to FSM transitions

**C++ side (generated by CXX, not by quent-codegen):**

- `.rs.h` headers that CXX produces from the bridge definitions
- The C++ application includes these headers directly
- No additional C++ code generation by quent-codegen is needed

## Build integration

Uses Corrosion (CMake ↔ Cargo integration):

- `corrosion_import_crate()` imports the Rust instrumentation crate
- Rust `build.rs` runs `cxx_build::bridges()` to generate C++ headers
- Generated headers are placed in a stable include directory for the C++ build

This pattern is already proven in the Sirius project.

## Rationale

### No wire format needed

C++ and Rust run in the same process. There is no network or process boundary
to cross. Serializing events on the C++ side and deserializing on the Rust side
would add overhead and complexity for no benefit.

### No separate C++ runtime

The Rust `Context` already handles event buffering, async export, and exporter
management. Reimplementing this in C++ would be duplication. The CXX bridge
gives C++ direct access to the Rust runtime with minimal overhead.

### CXX over raw C ABI

CXX provides type-safe bindings with automatic lifetime management
(`rust::Box`, `rust::String`, shared structs). A raw `extern "C"` API would
require manual memory management and lose type safety at the boundary. CXX also
generates the C++ headers, so both sides are guaranteed to match.

### Proven pattern

The Sirius project uses this exact architecture: CXX bridge, Rust observers
exposed to C++, Corrosion for build integration. The codegen automates what
Sirius implements by hand.

## Tradeoff

The C++ application must build the Rust crate as part of its build (via
Corrosion or equivalent). This adds a Rust toolchain dependency to the C++
build. For projects already using Quent (which is Rust-native), this is not an
additional burden.
