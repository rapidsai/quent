# Phase 2: Extract @quent/utils - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract five utility modules into `ui/packages/@quent/utils/` and migrate all app imports to use the package. The existing `~quent/types` path alias is removed. No new functionality is added — pure extraction + import migration.

**Delivers:** `cn()`, Rust-generated TypeScript types, `parseJsonWithBigInt`, color utilities, and formatter utilities — all accessible via `import { … } from '@quent/utils'`.

**Out of scope:** Any extraction from `api.ts` beyond `parseJsonWithBigInt` (that's Phase 3), Jotai atoms, React components, TanStack Query hooks.

</domain>

<decisions>
## Implementation Decisions

### Type re-export strategy
- **D-01:** Rust-generated types live in `examples/simulator/server/ts-bindings/` (52 files). `@quent/utils` re-exports them via a barrel using relative path imports. The `examples/simulator/server/ts-bindings/` directory remains the source of truth (generated by Rust/ts-rs).
- **D-02:** The `~quent/types` path alias is removed from `vite.config.ts` and `tsconfig.json` (if present). App code migrates from `import type { Foo } from '~quent/types/Foo'` to `import type { Foo } from '@quent/utils'`.
- **D-03:** Publishability note: the relative path from the package to `examples/simulator/server/ts-bindings/` crosses package boundaries. This is acceptable for this milestone (design for publishability, don't publish). When publishing, types must be bundled or copied — that's a Phase V2/publish concern.

### Color utilities
- **D-04:** All of `ui/src/services/colors.ts` is extracted to `@quent/utils`, including browser-specific canvas pattern functions (`createStripePattern`, `createDotPattern`, `createCrosshatchPattern`).
- **D-05:** Canvas pattern functions are annotated with `@remarks Browser-only — requires `document.createElement('canvas')`. Do not call in SSR or Node.js environments.` in JSDoc.
- **D-06:** Module-level mutable state (`colorAssignments`, `usedIndices`, `activePalette`) is preserved as-is. The Phase 1 `resolve.dedupe` config ensures a single module instance at runtime; no singleton risk.

### parseJsonWithBigInt extraction
- **D-07:** `parseJsonWithBigInt` is extracted from `ui/src/services/api.ts` and exported from `@quent/utils`.
- **D-08:** After extraction, `ui/src/services/api.ts` imports `parseJsonWithBigInt` from `@quent/utils`. This establishes the `@quent/client` → `@quent/utils` dependency direction for Phase 3 (Phase 3 will fully extract `api.ts` into `@quent/client`).

### Formatter utilities
- **D-09:** All of `ui/src/services/formatters.ts` is extracted to `@quent/utils`. The formatter imports of Rust types (`PrefixSystem`, `QuantitySpec`, `CapacityKind`) are resolved through the `@quent/utils` types barrel (same package — no cross-package import needed).

### cn() utility
- **D-10:** `cn()` is extracted from `ui/src/lib/utils.ts` and exported from `@quent/utils`. Dependencies `clsx` and `tailwind-merge` become direct `dependencies` (not peerDependencies) of `@quent/utils` since they are implementation details, not peer contracts.

### App import migration
- **D-11:** All app imports of the extracted modules are migrated in this phase:
  - `@/lib/utils` → `@quent/utils` (for `cn`)
  - `~quent/types/*` → `@quent/utils` (for Rust types)
  - `@/services/formatters` → `@quent/utils`
  - `@/services/colors` → `@quent/utils`
  - `parseJsonWithBigInt` from `@/services/api` → `@quent/utils` (within `api.ts` itself — already handled by D-08)
- **D-12:** After migration, `ui/src/lib/utils.ts` and `ui/src/services/colors.ts`, `ui/src/services/formatters.ts` are deleted (or reduced to re-exports from `@quent/utils` for a transition period — Claude's discretion on which approach is cleaner).

### Package dependencies
- **D-13:** `@quent/utils/package.json` dependencies: `clsx`, `tailwind-merge` as direct deps. No React, Jotai, or TanStack deps. No peerDependencies needed.
- **D-14:** `@quent/utils` has no package-level peerDependencies — it's a zero-framework-dep utility package. This is a requirement of its publishability design.

### Claude's Discretion
- Whether to delete `ui/src/lib/utils.ts`, `ui/src/services/colors.ts`, `ui/src/services/formatters.ts` or convert them to thin re-exports during migration. Either is fine — prefer deletion if no other app code is in those files.
- The exact structure of `src/index.ts` barrel (grouped by category or flat list) — whatever is clearest.
- Whether to break the extraction into multiple plan files or handle in one plan.

</decisions>
